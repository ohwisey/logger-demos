Build all of this in one go:
1. An add form at the top: service name, price, billed monthly, yearly or
   weekly, and the next renewal date. Each subscription is a row underneath
   showing its name, how it is billed, and its price, with a delete button
   that asks are-you-sure first.
2. One big number at the top: what all of it costs per month. Yearly prices
   divide by 12, weekly prices multiply by 52 then divide by 12, so the math
   is honest. Under it, a smaller line with the cost per year. Both update
   the moment anything changes.
3. A "coming up" list: every subscription renewing in the next 30 days,
   soonest first, showing when it lands (today, tomorrow, in 5d) and what it
   will charge. Renewal dates roll forward automatically once they pass, so
   I never re-type a date.
4. The warnings: any row renewing within 5 days gets highlighted, and each
   subscription can take an optional "free trial ends" date: when that date
   is within two weeks, show a clear line telling me to cancel before it
   charges me.

My subscription tracker is done. Now plug it into my Vitality dashboard (from the setup video).

First find my dashboard: it is my own copy of vitality-base. If the folder is
not on this computer, ask me for the GitHub link to my copy and clone it here
first, then continue.

Make a dashboard copy of my tracker as ONE self-contained HTML file:
1. Everything inline, no outside requests. It runs sealed inside a dashboard
   tile, so anything not in the file will not load.
2. No localStorage in this copy. The dashboard provides a bridge: save with
   await window.Vitality.save(data) and read it back with
   await window.Vitality.load(). You never write the bridge yourself.
3. If window.Vitality is missing (the file opened on its own), fall back to
   saving on the device, so the same file works both places.
4. Same theme, same features. Do not dumb anything down.

Save it as public/tiles/finance.html inside the dashboard folder. Before
pushing, check I am logged in to GitHub (gh auth status or a test push): if I
am not, walk me through gh auth login first (browser flow, one time). Then
commit and push.

Then PROVE it landed instead of hoping: ask me for my dashboard's live
address (the vercel.app one), and check /tiles/finance.html every
20 seconds for up to 4 minutes until it stops being a 404. The moment it is
live, tell me to refresh my dashboard: the Finance tile is now my tracker. If
it is still missing after 4 minutes, the deploy never ran: walk me through
vercel.com -> my project -> Deployments to see whether the newest push built
or failed, and fix what we find together.

When I am logged in, retry the push. This is a one-time setup: every push
after this just works.

My Finance tile is not right after the link-up (blank, not saving, or looks
off). Open public/tiles/finance.html in my dashboard folder and repair it.
Check, in order:
1. No localStorage anywhere in this file. Every save and load goes through
   await window.Vitality.save(data) and await window.Vitality.load().
2. The file is fully self-contained: no outside links, fonts, or requests.
3. The fallback exists: if window.Vitality is missing, it saves on the device.
4. The file sits at exactly public/tiles/finance.html, nowhere else.
Fix what you find, commit, push, and tell me in one line what was wrong.
