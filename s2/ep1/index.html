The base

This first prompt sets the whole look. Every prompt after inherits it, so your logger comes out on-theme without repeating yourself.

Why this prompt is longer

It carries the theme passport: the exact colors, font, and feel. Set it once here and every later prompt stays two lines. This is the trick that makes the whole build look designed instead of default.

0The single-file shelltheme passport · offline · saves in the browser 
Build a workout app as ONE single index.html file. No build step, works offline,
saves everything in the browser so it is still there when I reload.

THE THEME. Follow this exactly, for everything we add from now on:
- Background pure black (#000). No grays, no gradients on the page itself.
- One accent: mint (#6EE7B7). Mint is only for wins, live numbers, and the
  primary action. Never decorate with it.
- Two fonts. Headings, exercise names, and little action words in an elegant
  serif: "Instrument Serif" from Google Fonts, italic where it feels editorial.
  Everything else Inter: numbers bold and big, labels small and quiet.
- Set numbers as small serif roman numerals (i, ii, iii), muted.
- Buttons are pills (fully rounded). Small actions read as lowercase serif
  italic words: log, swap, chart, history.
- Big rounded cards (18px), hairline borders (white at 8%), lots of breathing room.
- Mobile first. Big tap targets, everything reachable with a thumb.
- First-use screens are never blank: one warm inviting line and one clear mint
  action, always.
- The overall feel: a premium Apple Health app crossed with an editorial
  magazine. Calm, spacious, one thing at a time.

For now just build the shell: a clean title bar and an empty "today" area below it.
Nothing else yet. We add one piece at a time.

One paste · same chat, straight after Part 3
My logger is done. Now plug it into my Vitality dashboard from Episode 0.

First find my dashboard: it is my own copy of vitality-base. If the folder is
not on this computer, ask me for the GitHub link to my copy and clone it here
first, then continue.

Make a dashboard copy of my logger as ONE self-contained HTML file:
1. Everything inline, no outside requests. It runs sealed inside a dashboard
   tile, so anything not in the file will not load.
2. No localStorage in this copy. The dashboard provides a bridge: save with
   await window.Vitality.save(data) and read it back with
   await window.Vitality.load(). You never write the bridge yourself.
3. If window.Vitality is missing (the file opened on its own), fall back to
   saving on the device, so the same file works both places.
4. Same theme, same features. Do not dumb anything down.

Save it as public/tiles/train.html inside the dashboard folder, commit, and
push. Then tell me to wait a minute for the deploy and refresh my dashboard:
the Train tile is now my logger.
L0If the push fails (GitHub login)one-time fix · 60 seconds 
My git push failed with an authentication error (git cannot log in to
GitHub). Fix it with me: check if the GitHub CLI (gh) is installed and
install it if missing, then walk me through gh auth login step by step
(GitHub.com, HTTPS, login with browser) and tell me exactly what to click.
When I am logged in, retry the push. This is a one-time setup: every push
after this just works.
L1If the tile acts upthe repair paste · 30 seconds 
My Train tile is not right after the link-up (blank, not saving, or looks off).
Open public/tiles/train.html in my dashboard folder and repair it. Check, in
order:
1. No localStorage anywhere in this file. Every save and load goes through
   await window.Vitality.save(data) and await window.Vitality.load().
2. The file is fully self-contained: no outside links, fonts, or requests.
3. The fallback exists: if window.Vitality is missing, it saves on the device.
4. The file sits at exactly public/tiles/train.html, nowhere else.
Fix what you find, commit, push, and tell me in one line what was wrong.
